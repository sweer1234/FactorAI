# FactorAI 节点库与官方模板仿作开发蓝图（V2）

> 目标：以“节点库 + 官方模板”为核心，对齐截图中的 AI 工作流产品形态，完成可运行、可复用、可扩展的高保真版本。

---

## 1. 背景与目标

当前系统已具备前后端闭环（工作流管理、运行、报告），但与目标形态仍有差距：

1. **节点库深度不够**：节点数量、分组层级、节点说明文档、参数面板交互尚不完整。  
2. **官方模板不足**：模板数量与结构复杂度低，缺少“可直接运行的成体系案例”。  
3. **编辑器体验差距**：节点侧栏（节点库/日志/当前节点）、右侧节点说明、历史工具栏等关键体验需补齐。  

本阶段聚焦：**优先仿作节点库与官方模板能力**，并确保“可执行”而非仅界面展示。

## 4. 节点规范补充（V2）

## 4.1 节点元数据（NodeSpec）

```json
{
  "id": "ml.xgboost",
  "name": "XGBoost模型",
  "category": "03-机器学习",
  "version": "1.0.0",
  "inputs": ["feature_df", "label_df"],
  "outputs": ["pred_df", "feature_importance"],
  "params_schema": {},
  "ui_schema": {},
  "doc_markdown": "...",
  "example_graph": {},
  "runtime": {
    "engine": "python",
    "timeout_sec": 120,
    "memory_mb": 1024
  }
}
```

## 4.2 节点文档面板标准

每个节点必须提供：

- 节点介绍（用途）
- 工作流示例（建议配缩略图/最小链路）
- 核心参数（必填/默认值/范围）
- 输入字段（类型 + 必填字段）
- 输出字段（类型 + 结构约束）
- 注意事项（常见报错与规避方式）

## 4.3 节点执行一致性要求

- 输入输出 DataFrame 规范统一（`date/symbol` 约束）
- 输出字段 schema 校验失败即终止任务
- 每节点产出中间产物（用于“当前节点”调试查看）
- 记录耗时、输入规模、异常堆栈

## 6. 模板结构标准

每个模板由以下组成：

1. 模板元数据（名称、标签、分类、说明、更新时间）
2. DAG 图定义（nodes/edges/layout）
3. 参数预设（含默认训练区间、回测区间、手续费）
4. 输入数据契约（数据表要求）
5. 预期输出（报告指标、可视化面板）

示例：

```json
{
  "template_id": "tpl-official-06",
  "name": "官方案例-6-超参搜索",
  "graph": {},
  "default_params": {
    "train_start": "20240101",
    "train_end": "20250301",
    "fee_bps": 4
  }
}
```

---

## 10. 验收标准（必须达成）

1. 节点库分类、数量、层级与截图要求一致（允许视觉风格差异）
2. 官方案例模板不少于 10 个，且可一键运行成功率 >= 90%
3. 每个节点都有“介绍 + 参数 + 输入输出 + 注意事项”
4. 编辑器支持节点定位、当前节点查看、日志追踪
5. 模板运行后均可生成报告（至少含 IC、收益、回撤、分层）

---

## 11. 风险与应对

1. **节点数量大，研发成本高**  
   - 方案：先做节点骨架 + 参数校验，算法逻辑分批实装

2. **模板运行依赖数据质量**  
   - 方案：先统一样本数据契约，补充数据检查节点

3. **Python 节点安全风险**  
   - 方案：持续收紧沙箱白名单，引入执行审计
